微服务

什么是微服务架构，为什么采用它

  • 微服务是一种软件设计的架构风格，它解决了传统单体应用的缺点。通常人们会根据业务逻辑或者业务特点将一个单一应用程序拆分成一组小型服务，服务之间通过 HTTP 形式的 RESTful API 进行通信
  • 我们采用微服务架构主要从两个方面考虑：第一，我们希望某个业务模块发生迭代更新时，不需要重新构建和部署整个服务；第二，我们希望每个模块由不同的团队负责，对开发进行提速提效
  • 除了我们考虑的方面，微服务还能做到不同服务根据场景的不同采用不同的技术方案，提高了灵活性
  • 微服务也存在一些弊端
    • 服务之间的测试不太好覆盖，这增加了 Bug 定位的难度
    • 增加了运维的复杂性，需要对多个服务进行监督
    • 对网络通信要求比较高，需要考虑通信时延和网络开销。

单体应用

  • 在单体应用的架构风格下，软件的所有功能模块和业务逻辑都放在单个服务中
  • 放在单个服务的优点是十分明显的，即部署方便，运行/测试简单
  • 但随着业务量的增加，项目的代码会变得复杂臃肿，维护困难，而且当系统发生变更时，整个服务都需要重新构建和部署，增加了系统的停机时间，效率十分低下。

拆分和集群之间有什么区别

  • 拆分：不同的多台服务器上部署不同的服务模块，模块之间通过 RPC 通信和调用，用于拆分业务功能，独立部署，多个服务器共同组成一个整体对外提供服务
  • 集群：不同的多台服务器上部署相同的服务模块，通过分布式调度软件进行统一调度，用于分流容灾，降低单个服务器的访问压力

微服务在你们项目中是怎么应用的

  • 在 SCA 中，我们主要拆分成 5 个服务
    • backend_service：它主要充当 gateway 的作用（流转 data_service 和 receive_service 的数据），也承担部分数据接口
    • receive_service：它主要负责解析文件的接收和管理
    • data_service：主要负责数据解析工作
    • alert_service：主要负责报警信息处理、邮件发送
    • supplier_service：主要对供应商信息进行维护管理

RESTful 是啥，为什么需要使用它

  • RESTful 是一套接口设计规范
  • 在采用前后端分离的模式下，前后端是通过接口来完成通信的，随着业务的扩大和接口数量的增加，接口的可读性、扩展性也变得愈加重要，而 RESTful 满足这些要求（降低沟通成本，提高开发效率）
  • 在 RESTful 架构中，每个网址代表一种资源，因此网址中不能有动词，只能有名词，而且使用的这些名词往往和数据库的表格名称对应。一般来说，数据库中的表都是同种记录的集合，因此 API 中的名词也应该使用复数

你刚刚说在 RESTful 架构中，每个网址代表一种资源，那我对它进行操作的时候应该怎么做

  • 一般我们对资源会进行四种操作，即获取、创建、修改和删除，这四种操作刚好对应了 HTTP 协议提供的 GET、POST、PUT 和 DELETE 方法，这样，通过判断 HTTP 的请求方法，我们能清楚地知道我们需要对资源进行那种操作
  • 对资源的修改也可以使用 PATCH 方法，它跟 PUT 的区别是：PATCH 是用来对资源进行局部更新，即只需要传输需要修改的部分，减少带宽

RESTful

简介

  • RESTful API 是目前比较成熟的一套接口设计理论（规则）

  • 之所以出现这种模式，是因为当前发展趋势是前端设备层出不穷（比如说移动端各种 APP、小程序；电脑端的网页、应用程序等），但前后端之间的通信需要一种统一的机制，而 RESTful API 就是一种比较成熟的机制。

  • 在 RESTful 架构中，每个网址代表一种资源（resource），所以网址中不能有动词，只能有名词，而且所用的名词往往与数据库的表格名对应。一般来说，数据库中的表都是同种记录的”集合”（collection），所以 API 中的名词也应该使用复数。

  • 对于资源的具体操作类型，由 HTTP 动词表示

    • GET（SELECT）：从服务器取出资源（一项或多项）
    • POST（CREATE）：在服务器新建一个资源
    • PUT（UPDATE）：在服务器更新资源（客户端提供改变后的完整资源）
    • PATCH（UPDATE）：在服务器更新资源（客户端提供改变的属性）
    • DELETE（DELETE）：从服务器删除资源

  • RESTful API 最好做到 Hypermedia，即返回结果中提供链接，连向其他 API 方法，使得用户不查文档，也知道下一步应该做什么。

  • API 的身份认证应该使用 OAuth 2.0 框架

  • 服务器返回的数据格式，应该尽量使用 JSON，避免使用 XML。

  • RESTful 架构（总结）

    • 每一个 URI 代表一种资源
    • 客户端和服务器之间，传递这种资源的某种表现层
    • 客户端通过 HTTP 动词，对服务器端资源进行操作，实现”表现层状态转化”

进化顺序

  • RPC $\rightarrow$ SOAP $\rightarrow$ RESTful

OAuth2.0

  • 目前最流行的授权机制，用来授权第三方应用，获取用户数据。

    • 数据的所有者告诉系统，同意授权第三方应用进入系统，获取这些数据。系统从而产生一个短期的进入令牌（token），用来代替密码，供第三方应用使用。

  • Token 种类

    • 授权码（authorization-code）
    • 隐藏式（implicit）
    • 密码式（password）
    • 客户端凭证（client credentials）

  • 为了防止 token 不被滥用，第三方首先要到系统备案，拿到客户端 ID（client ID）和客户端密钥（client secret）
